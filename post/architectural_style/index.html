Pipe and Filter

  • Suitable for application that require a defined series of independent computations
  • Components reads streams of data and on its inputs and produces streams of data on its outputs

Components: called filters apply local transformation to their input streams and often do their computing incrementally so that the output begins before all input is consumed. Connectors: called pipes serve as conduits for the streams, transmitting outputs of one filter to inputs of another

Types:

  • Pipelines: restricts topologies to linear squences of filters
  • Batch Sequential: A degenerate case of a pipline architecture where each filter processes all of its input data before producing any output

Example:

  1. Unix Shell Scripts
  2. Traditional Compilers

Advantages:

  • Easy to understand
  • Support reuse
  • Easily Maintainable
  • Specialized analysis
  • Support concurrent execution

Disadvantages:

  • Not good for handling interactive systems
  • Loss of performance
  • Increased Complexity

Repository Style

  • Suitable for application in which the central issue is establishing, augmenting and maintaining a complex central body of information
  • Typically the information must be manipulated in a variety of ways. Long term persistence is required

Components: Central data structure representing the current state of the system. A collection of independent components that operate on the central data structure Connectors: Typically direct memory access

Specializations:

  1. Changes to the data structure / trigger computations.
  2. Data structure in memory. This is presistent option
  3. Data structure on disk
  4. Concurrent computation

Types:

  • Blackboard: the repository is active and can activate the components in response to changes in it self. Where as in repository style, the repository is inactive and just maintains the central data.

Examples:

  1. Central Code Repository Systems
  2. Programming Environments
  3. Graphical Editors
  4. Database Management Systems
  5. Games - multiple users editing and interacting with the same world

Advantages:

  • Efficient way to store large amounts of data
  • Sharing model is published as the repository schema
  • Centralized management

Disadvantages:

  • Must agree on a data model before
  • Difficult to distribute data
  • Data evolution is expensive
  • Single point of failure

Implicit invocation Sytle

  • Suitable for application where the creators do not need to know who or how many will need to use the data
  • A component broadcasts output that are consumed by one or more of the other components

Components:

  • Data Generators
  • Data Consumers

Connectors:

  • Event Bus
  • Procedure Calls

Publish - Subscribe Variant:

  • Subscribers register to receive specific messages
  • Publishers maintain a subscription list and broadcast messages to subscribers
  • Subscribers connect to publishers directly
  • Diagram is generally scattered between publishers and subscribers
  • Examples: Twitter / Environment Canada

** Event-Based Variant:**

  • Components asynchronously emit and receive “events” communicated over event bus
  • Components do not directly talk to each other
  • Diagram is gerally sequential between even bus and subscribers/publishers
  • Examples: Debugger breakpoint announcement / Editors scrolls to appropriate source line and hihglights it

Advantages:

  • Provides strong support for reuse
  • Eases system evolution
  • Effiecient dissemination of one way info

Disadvantage:

  • Need special protocols when number of subscribers is very large
  • When a component announces an event
    • It has no idea what other component will respond to it
    • It cannot rely on the order in which the reponses are invoked
    • It cannot know when responses are finished

Layered Style:

  • Architecture is seperated into ordered layers
    • A program in one layer may obtain services from a layer below it

Components: called layers, does local calculations and communicates with adjacent layers Connectors: are pre-determined protocols for communication between layers

Examples:

  • Layered Communication Protocals: Network/Link/Physical layer
  • Operating Systems

Advantages:

  • Design
  • Enhancement
  • Reuse

Disadvantages:

  • Not all systems are easily structured in a layer fasion
  • Performance requirements

Client-Server Style

  • Suitable for applications that involve distributed data and processing across a range of components

Components:

  • Clients
  • Servers

Connectors: The Network

Examples:

  • File Servers: Sharing files across the network
  • DB Servers:
    • More effient use of distributing power than file servers
    • Client passes SQL requests as messages to the DB servers; results are returned over the network

Advantages:

  • Data Distribution is straight forward
  • Tranparency of location
  • Mix and match heterogeneous platforms
  • Easy to add new servers or upgrade existing servers

Disadvantages:

  • No Central register of names and services – might be hard to find out what servifces are available

Process Control Style

  • Suitable for application whose purpose is to maintain specified properties of the output of the process at given reference values
  • Also called feedback control system

Components:

  • Process Definition
  • Control Algorithm

Connectors:

  • Process Variables
    • Controlled variable
    • Input variable
    • Manipulated variable
  • Set Point
  • Sensors

Example: Varios realtime systems (Neclear Power Plants, Car Cruise Control)

Serverless Architecture

  • Using and paying for functionality on an as neede bases

Components:

  • Function (Container)

Connector:

  • Event
    • Synchronous (HTTP request)
    • Asynchronous (Message Queue)

Advantages:

  • Pay per use
  • Less maintenance
  • Fast Deployment
  • Easiy to debug

Disadvantages:

  • 3rd party depenedency
  • Intial latency (cold start)
  • Stateless nature - won’t remember the previous state

Microservices Architecture

  • forcus on building single function modules with well defined interfaces and operations
  • Suitable for applications that require short release cycles and must be highly scalabale

Components: Self contained services

Connectors:

  • Synchronous: HTTP
  • Asynchronous - Advanced Messaging Queue Protocol

General Properties:

  • Single responsibility
  • Can be developed and maintained by single independent teams
  • it is not overly dependent on another service. If so, merge
  • It must be seperately deployable
  • It maintains the integrity and consistency of the data overall

Advantages

  • Independently deployable
  • Easy CI/CD integration
  • Independently Scalable
  • Reduce downtime
  • Team autonomy
  • Simple to maintain

Disadvantages:

  • expensive
  • more distributed complixity

MVVM - Model View View Model

  • Seperate the UI from business logic

Components:

  • View
  • ViewModel
  • Model

Connectors:

  • Events and Method Calls

Example:

  • Any RSS reader app or any app with UI and some data in the back to be manipulated

Advantages:

  • Abstraction helps in:
    • Testing
    • Different Teams work independently

DisAdvantage:

  • For simple applications MVVN can be overkill
  • Debugging

MVC vs MVP vs MVVM

  • Controller (C) - Connects Model with View but is not required to be organized
  • Presenter (P) - Gets the data from the Model and the UI rules from the view and applies to the data
  • ViewModel (VM) - Like P in that it is more organized and we know what data needs to be associated for a view but then seperates the actual presentation logic of the UI thus making it easier to test
